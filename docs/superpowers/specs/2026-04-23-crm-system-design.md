# CRM-система — Спецификация дизайна
**Дата:** 2026-04-23  
**Проект:** operator-dashboard  
**Этап:** 1 — CRM-ядро (задачи, сотрудники, команды)

---

## 1. Контекст и цель

Единое приложение, объединяющее существующий **дашборд мониторинга выручки** и новую **CRM для управления командой**. Реализуется поверх текущего репозитория `operator-dashboard` (React + Vite + Tailwind + Supabase, деплой на Vercel).

### Бизнес-контекст

Иерархия сущностей бизнеса:
```
Платформа (PRIME, UBA, DREAM, AFA)
  └─ Агентство (Club Monaco, Belowed, Kleopatra…)
    └─ Админ-панель (107, 2444, 3059…)
      └─ Оператор (переводчик, source_id из Tableau)
        └─ Клиенты (Trusted User ID из Tableau)
```

Сотрудники компании (менеджеры) управляют операторами и ведут отчётность через CRM.

## 2. Роли и доступ

Все сотрудники системы называются **«Сотрудники»**. Каждый имеет три независимых слоя:

| Слой | Описание | Пример |
|---|---|---|
| **Роль** | Тип сотрудника | `moderator` |
| **Права** | Что может делать | `create_tasks`, `create_users` |
| **Атрибуты** | Дополнительные настройки | `shift=ДЕНЬ`, `panel_id=2509` |

### 2.1 Роли

**Супер-Админ (`superadmin`)**
- Создаёт всех сотрудников и управляет их ролями/правами
- Подтверждает или отклоняет запросы на удаление сотрудников
- Видит всё — все задачи, всех сотрудников, всю аналитику
- Не ограничен никакими правами

**Администратор (`admin`)**
- Создаёт задачи для Модераторов и Тим Лидеров
- Может редактировать и перемещать задачи
- Видит выполненные и просроченные задания всех подчинённых
- Получает оповещения о просроченных заданиях
- Видит весь список сотрудников
- Назначает команды операторов тим-лидерам
- Может создавать сотрудников (если есть право `create_users`)
- Может запросить удаление сотрудника (с указанием причины, требует подтверждения Супер-Админа)

**Модератор (`moderator`)**
- Видит список задач от администратора
- Может добавлять собственные задачи (дополнительные)
- Загружает скриншоты и записи в задачи
- Управляет списком своих подчинённых (добавляет/убирает операторов)
- Видит список своих операторов с их статусами
- Атрибут: смена (`ДЕНЬ` / `ВЕЧЕР` / `НОЧЬ`)

**Тим Лидер (`teamlead`)**
- Видит список задач от администратора
- Может добавлять собственные задачи (дополнительные)
- Загружает скриншоты и записи в задачи
- Управляет составом своих команд операторов
- Команды назначает администратор
- Видит список операторов своих команд

**Оператор (`operator`)**
- Видит свою статистику выручки (только свои данные)
- Видит своё место в топе операторов
- Видит задачи, поставленные по нему (как объект задачи)
- В будущем: внутренний чат
- Не создаёт задачи, не управляет другими сотрудниками

### 2.2 Система прав (permissions)

Права хранятся отдельно от роли — каждый флаг можно дать или отозвать вручную:

| Право | Описание | Роль по умолчанию |
|---|---|---|
| `create_users` | Создавать сотрудников | superadmin, admin (опционально) |
| `manage_roles` | Назначать роли и права | superadmin |
| `create_tasks` | Создавать задачи | superadmin, admin, moderator, teamlead |
| `view_all_tasks` | Видеть задачи всех сотрудников | superadmin, admin |
| `view_own_tasks` | Видеть только свои задачи | moderator, teamlead, operator |
| `view_all_revenue` | Полный дашборд выручки | superadmin, admin |
| `view_own_revenue` | Только своя статистика | operator |
| `view_team_revenue` | Выручка своей команды/смены | moderator, teamlead |
| `send_reminders` | Кнопка «Напомнить» по просрочке | superadmin, admin |
| `manage_teams` | Управлять командами | superadmin, admin, moderator, teamlead |
| `use_chat` | Внутренний чат (будущее) | все |

### 2.3 Атрибуты сотрудников

Гибкое хранилище — ключ/значение. Расширяется без изменения схемы:

| Ключ | Значение | Кому |
|---|---|---|
| `shift` | `ДЕНЬ` / `ВЕЧЕР` / `НОЧЬ` | Модераторам |
| `panel_id` | `2509`, `107`… | Модераторам |
| `team_id` | UUID команды | ТЛ, Операторам |
| `tableau_id` | source_id из Tableau | Операторам |

### 2.4 Реферальные коды

Автогенерируются при создании сотрудника, никогда не меняются:

```
Администратор:  ADM-ИванП-001
Модератор:      MOD-ЛераД-002
Тим Лидер:      TL-АннаМ-003
Оператор:       OP-ВадимС-012
```

Правила: Имя полностью (заглавная), Фамилия первая буква (заглавная), порядковый номер трёхзначный уникальный в пределах роли. Дополнительно можно задать псевдоним (алиас) — отображается рядом с кодом.

## 3. Система задач

### 3.1 Статусы

| Статус | Описание |
|---|---|
| `assigned` | Назначена, срок не наступил |
| `in_progress` | Исполнитель открыл задачу |
| `done` | Выполнена (обязателен комментарий или вложение) |
| `overdue` | Срок прошёл, статус не «Выполнена» |

После перехода в `done` задача блокируется от редактирования. Удалить задачу нельзя.

### 3.2 Пулы задач

**Модератор (по дате):**
- Сегодня
- Завтра
- Эта неделя
- Просроченные

**Тим Лидер (по неделям):**
- Эта неделя
- Следующая неделя
- Просроченные

### 3.3 Кто кому ставит задачи

| Кто ставит | Кому |
|---|---|
| Супер-Админ | Всем |
| Администратор | Модераторам, ТЛ, Операторам |
| Модератор | Своим операторам, другим модераторам |
| Тим Лидер | Операторам своей команды, другим ТЛ |

### 3.4 Шаблоны обязательных заданий модератора

Предустановленные типы задач с повторяемостью:

| Шаблон | Периодичность | Вложения |
|---|---|---|
| Вычитка чатов | Раз в неделю (каждый переводчик) | Скриншоты |
| Вычитка бустера | Раз в неделю (каждая команда) | Скриншоты |
| Индивидуальный звонок | Раз в неделю (каждый переводчик, 1–1.5 ч) | Запись звонка |
| Еженедельный отчёт по балансам | Раз в неделю | Текст + скриншоты |
| Отчёт по балансам после смены | Каждая смена | Текст |
| Пересменка | При передаче смены | Текст, опционально скриншоты |
| Другое (произвольная) | — | Любые |

**Шаблоны ТЛ:**

| Шаблон | Периодичность | Вложения |
|---|---|---|
| Вычитка тепловой таблицы | Раз в неделю | Скриншот таблицы + комментарий |
| Вычитка премиума | Раз в неделю | Скриншоты |
| Созвон с командой | Раз в неделю | Запись + тезисы |
| Другое (произвольная) | — | Любые |

Администратор может добавлять новые типы задач.

### 3.5 Вложения (Supabase Storage)

| Тип | Форматы | Лимит |
|---|---|---|
| Скриншоты | JPG, PNG, WEBP | до 10 МБ / файл, до 20 файлов |
| Запись звонка | MP3, MP4, OGG, M4A | до 200 МБ, 1 запись |
| Документы | PDF, DOCX | до 20 МБ / файл, до 5 файлов |

Хранение: Supabase Storage. Signed URL с TTL 1 час. В БД — только метаданные.

## 4. Структура команд

- **Оператор** → входит в команду ТЛ и/или список модератора
- **Модератор** → управляет своим списком операторов (добавляет/убирает напрямую)
- **Тим Лидер** → управляет составом команды; команды назначает Администратор
- Оператор без привязки → пул «Без команды»

---

## 5. Workflow удаления сотрудника

1. Администратор нажимает «Удалить», вводит причину → запрос уходит со статусом `pending`
2. Супер-Админ видит запрос в разделе «Оповещения»
3. **Подтверждает** → сотрудник деактивируется (`is_active = false`), история сохраняется, тег `[неактивен]`
4. **Отклоняет** → сотрудник остаётся активным, администратор получает уведомление с причиной отказа

Деактивированный сотрудник не может войти. Все данные сохраняются.

## 7. Клиенты (данные, без аккаунтов)

Клиенты — end-users платформы, закреплённые за операторами. В CRM отображаются как данные:
- Просматриваются в карточке оператора
- Linked по `trusted_user_id` из Tableau
- Аккаунтов в CRM не имеют, не логинятся

## 8. Схема базы данных (Supabase)

### Расширяемые таблицы

**`users`** — расширяется новыми полями:
```sql
id            uuid         PK
ref_code      text         -- MOD-ИванП-001 (auto, immutable)
first_name    text
last_name     text
alias         text         -- псевдоним (опционально)
role          enum         -- superadmin|admin|moderator|teamlead|operator
email         text
password_hash text
timezone      text
tableau_id    text         -- source_id из Tableau (для операторов)
is_active     bool
created_at    timestamptz
created_by    uuid → users
```

### Новые таблицы

**`user_permissions`**
```sql
user_id       uuid → users
permission    text         -- create_users, create_tasks…
granted_by    uuid → users
granted_at    timestamptz
```

**`user_attributes`**
```sql
user_id       uuid → users
key           text         -- "shift", "panel_id"…
value         text         -- "ДЕНЬ", "2509"…
```

**`tasks`**
```sql
id            uuid         PK
title         text
type          text         -- шаблон или произвольная
assigned_to   uuid → users -- исполнитель
related_to    uuid → users -- о ком задача (оператор, опционально)
created_by    uuid → users
due_date      timestamptz
status        enum         -- assigned|in_progress|done|overdue
comment       text
created_at    timestamptz
completed_at  timestamptz
```

**`task_attachments`**
```sql
id            uuid         PK
task_id       uuid → tasks
file_name     text
file_type     text
file_size     bigint
storage_path  text         -- путь в Supabase Storage
uploaded_by   uuid → users
uploaded_at   timestamptz
```

**`teams`**
```sql
id            uuid         PK
name          text
leader_id     uuid → users -- Тим Лидер
created_by    uuid → users -- Администратор
created_at    timestamptz
```

**`team_members`** — операторы в команде ТЛ
```sql
team_id       uuid → teams
user_id       uuid → users
added_by      uuid → users
added_at      timestamptz
```

**`moderator_operators`** — операторы в списке модератора (независимо от команды ТЛ)
```sql
moderator_id  uuid → users
operator_id   uuid → users
added_by      uuid → users
added_at      timestamptz
```

**`notifications`**
```sql
id            uuid         PK
recipient_id  uuid → users
type          text         -- task_reminder|deletion_request|deletion_approved…
payload       jsonb        -- контекст уведомления
is_read       bool
created_at    timestamptz
```

**`deletion_requests`**
```sql
id            uuid         PK
target_user   uuid → users -- кого удаляют
requested_by  uuid → users -- кто запросил
reason        text
status        enum         -- pending|approved|rejected
reviewed_by   uuid → users
review_note   text
created_at    timestamptz
reviewed_at   timestamptz
```

### Существующие таблицы — без изменений

- `hourly_revenue` — (refcode, date, hour, delta) — данные из Tableau
- `snapshot` — (refcode, client_id, cumulative) — кумулятив

Связка: `users.tableau_id = hourly_revenue.refcode`

## 11. Технический стек

- **Frontend:** React 19 + Vite + Tailwind CSS v4 + shadcn/ui
- **Backend:** Supabase (PostgreSQL + RPC функции + Storage)
- **Auth:** Кастомная (localStorage сессия, JWT через RPC), 8 часов
- **Файлы:** Supabase Storage, signed URL TTL 1 час
- **Деплой:** Vercel (текущий)
- **UI-компоненты:** shadcnblocks Premium через MCP (`.mcp.json`) — блоки и готовые дашборды как референсы. Без промежуточного этапа Figma.
- **Иконки:** Lucide (SVG). Никаких эмодзи в интерфейсе.

## 12. Адаптивность (Mobile-first Responsive)

Критическое требование — значительная часть сотрудников будет работать с CRM с телефона (модераторы на смене, ТЛ, операторы).

**Подход:** mobile-first responsive — один код, адаптирующийся к экрану.

**Breakpoints (Tailwind defaults):**
- `base` (< 640px) — мобильный
- `sm` (640px+) — большой телефон / маленький планшет
- `md` (768px+) — планшет
- `lg` (1024px+) — десктоп
- `xl` (1280px+) — широкий десктоп

**Правила адаптации:**
- **Навигация:** Sidebar на desktop → bottom nav или drawer на mobile
- **Таблицы:** На десктопе — полная таблица. На мобильном — карточки (card-per-row). Приоритетные колонки остаются.
- **Фильтры:** На десктопе inline. На мобильном — в выезжающей панели (Sheet).
- **Модальные окна:** Desktop dialog → mobile full-screen sheet.
- **Детали задачи:** Desktop split-view (список + правая панель). Mobile — отдельная страница / full-screen sheet.
- **Формы:** Одна колонка на мобильном, две на планшете/десктопе.
- **Кнопки действий:** Минимум 44×44px (touch target).

**Что тестируем на мобильном (каждую фичу):**
- Создание задачи с вложением
- Отметка задачи выполненной
- Загрузка фото с камеры / галереи
- Прослушивание записи звонка
- Навигация между пулами задач

**Офлайн / слабая связь:** В Этапе 1 не требуется, но UI должен корректно показывать состояние загрузки и ошибки сети.
