# Subplan 3 · Clients (Модели) — фидбэк по дизайну

**Контекст для Claude Design**: ниже — структурированный фидбэк по макетам Subplan 3 (master-detail для раздела «Клиенты / Модели»). Просмотрены все вкладки: `List`, `Detail · Фото`, `Detail · Видео`, `Create slide-out`, `Empty · zero`, `Empty · filter`, `Loading`, `Upload progress`, `Drop overlay`, `Lightbox`, `Video error`.

Общая оценка: макеты на хорошем уровне, основные паттерны (master-detail, slide-out для создания, masonry для фото, fixed-grid для видео, drag-to-reorder, drop-overlay, lightbox) выбраны правильно. Фидбэк ниже — про доработку states, edge cases, согласованность между экранами и микрокопию.

---

## 0. Терминология (сквозное правило — применить ко всем экранам)

Раздел в навигации и breadcrumb называется **«Клиенты»**, но внутри страницы используется **«Модели»** (page title `Модели 9`, CTA `+ Добавить модель`, empty `Моделей пока нет` и т.д.). Это создаёт когнитивный диссонанс — пользователь не понимает, это две разные сущности или одно и то же.

**Решение продукта**: использовать **«Клиенты» / «клиент»** везде. Термин «модель» убрать из UI.

**Что меняется** (полный список переводов):

| Было | Стало |
|---|---|
| `Модели 9` (page title) | `Клиенты 9` |
| `+ Добавить модель` | `+ Добавить клиента` |
| `Моделей пока нет` | `Клиентов пока нет` |
| `Здесь появятся клиенты-модели агентства. Каждая модель закреплена за платформой и агентством.` | `Здесь появятся клиенты агентства. Каждый клиент закреплён за платформой и агентством.` |
| `+ Добавить первую` | `+ Добавить первого` |
| `Выберите модель слева` | `Выберите клиента слева` |
| `Профиль, фото- и видео-галерея откроются в этой панели.` (ок, без сущности) | без изменений |
| `Новая модель` (slide-out title) | `Новый клиент` |
| `Поля со звёздочкой обязательны` | без изменений |
| `Аватар · отдельно от галереи` | без изменений |
| `Создать модель` (slide-out CTA) | `Создать клиента` |
| `Загружается список моделей…` (loading slow) | `Загружается список клиентов…` |
| `Поиск по name / alias…` (ок, без сущности) | без изменений |
| `Активные 7` (фильтр статуса) | без изменений |
| Toggle `Активна` в шапке профиля | без изменений (статус, не сущность) |

**Применимость**: правило действует для всего Subplan 3. На уровне CRM в целом — рассмотреть как cross-cutting (см. `_cross-cutting.md`, когда будет создан).

## 1. Empty · zero

**Что хорошо**: dashed-border карточка пустого состояния с иконкой `user-plus`, заголовком, поясняющим копирайтом и вторичным CTA `+ Добавить первую`. Правая панель показывает empty-hint про master-detail.

**Правки:**

- **Дублирующиеся CTA**: `+ Добавить модель` в шапке + `+ Добавить первую` в карточке. Для zero-state допустимо, но в `Empty · filter` второй CTA должен превращаться в `Сбросить фильтры` (см. ниже — там уже исправлено правильно).
- **Счётчик `Активные 7` рядом с «0 моделей»** — выглядит противоречиво. Если это реальный mock, нужно проверить логику счётчиков в фильтрах: они должны считаться по текущему набору, а не по всей базе.
- **Правая панель empty-hint** в split-режиме на узких экранах занимает много места — рассмотреть схлопывание правой панели для viewport <1280px.

## 2. Empty · filter

**Что хорошо**: чёткая иерархия счётчиков (`Модели 9` всего ≠ 0 подходит), активный фильтр визуально выделен (синяя пилюля с ✕), CTA в карточке корректно меняется на `Сбросить фильтры`, иконка `funnel-x`, микрокопия называет конкретный фильтр.

**Правки:**

- **Дублирование путей сброса**: ✕ на пилюле = `Сбросить фильтры` когда фильтр один. Когда фильтров 2+, текст должен меняться: «Снимите один из фильтров или сбросьте все 3».
- **Состояния фильтров**: пилюля «Активные 7» рядом с активной «Агентство: Hex Talent» имеет одинаковый visual weight с нейтральной «Платформа все». Нужны три состояния: default / value-set / changed-by-user. Только последнее выделять как primary (синий).
- **`+ Добавить модель`** в шапке остаётся primary — рассмотреть понижение до secondary в filter-empty (primary должен быть `Сбросить фильтры`).
- **Правая панель** в filter-empty: либо схлопнуть, либо синхронизировать копирайт («Сначала найдите модель в списке»).
- **Search-empty отличается от filter-empty**: если в поиске «alex» — копирайт «По запросу „alex" в Hex Talent ничего не найдено», и сброс search ≠ сброс фильтров (отдельные действия). Сейчас mock’а нет.

## 3. Create slide-out (форма «Новая модель»)

**Что хорошо**: slide-out (а не модалка) — правильный выбор для master-detail. Поля сгруппированы логично, микрокопия конкретная под каждым полем, `Esc закрыть` виден в футере, avatar отдельно от галереи (защита от частой путаницы), placeholder’ы наводящие.

**Правки:**

- **CAPS LOCK на всех label’ах** (`NAME`, `ALIAS`, `PLATFORM`, `AGENCY`, `DESCRIPTION`, `TABLEAU ID`) — при 6+ полях uppercase визуально шумит. Рассмотреть sentence case с muted-foreground (Linear/Vercel-стиль).
- **«Markdown-разметка не поддерживается»** — negative hint. Переформулировать в позитив: «Plain text. Перенос строки сохраняется», или убрать совсем.
- **`TBL-0000` placeholder** выглядит как валидное значение. Лучше `например, TBL-2351` или `TBL-...`.
- **Platform / Agency в одну строку** на ширину slide-out (~480px) — длинные названия типа «Hex Talent Management» обрежутся. Поставить колонкой или проверить overflow.
- **Avatar drop-zone** — если поддерживает drag-and-drop, добавить «или перетащите файл». Если нет — игнорировать.
- **Confirm на закрытие при грязной форме** не виден на скрине. Esc + Отмена должны спросить подтверждение если есть несохранённый ввод.
- **Состояния валидации** не показаны — нужен mock «Empty submit / Validation error» с inline-ошибками под required-полями.
- **Loading state формы** при отправке: `Создаётся…` со спиннером, форма disabled.
- **Tableau ID — единственная внешняя интеграция**. Если планируются ещё (Notion/Linear) — обернуть в секцию «Опционально / Интеграции» с заголовком.

## 4. List + Профиль (Detail)

**Что хорошо**: левый список — сильный паттерн (avatar, имя+alias, мета-строка `платформа · агентство`, счётчик файлов). Архивные модели приглушены и помечены пилюлей `АРХИВ`. Шапка профиля компактная, инфо-чипы на одной строке. Правая колонка `Активность` / `Сводка` — отличная идея, особенно `Система создала клиента TBL-0341`. Хинт `click → inline edit` тонкий и понятный. Read-only поля помечены явно. Toggle `Активна` в шапке для быстрого статуса.

**Правки:**

- **Дублирование счётчика файлов**: `142 файлов` в шапке + `Медиа · Фото 118 + Медиа · Видео 24` в табах + `Сводка: Фото 118 / Видео 24` справа. Три раза одно. Убрать счётчик из шапки (табы и так несут эту инфу).
- **Грамматика плюрализации**: `142 файлов` → `142 файла`. Проверить формы: 1 файл / 2-4 файла / 5+ файлов / 21 файл / 22 файла.
- **Двухколоночный лейаут «Поля профиля»**: NAME / ALIAS, PLATFORM / AGENCY — короткие, попарно, как в форме создания.
- **`AVATAR` как поле** в списке полей профиля — это файл, не метаданные. Либо убрать (avatar в шапке достаточно), либо вынести в карточку «Файлы профиля».
- **`СОЗДАНО · 12 фев 2026 · Системой`** дублирует событие в `Активность`. Убрать из карточки полей.
- **`Активность` показывает 3 события** без «Показать все». Нужна ссылка `Показать всё (12)` или scroll внутри карточки.
- **`Сводка`** сейчас — только Фото/Видео (дубль табов). Либо расширить (просмотры, доход, посты), либо убрать.
- **`‹ ›` next/prev model** — добавить tooltip с именем следующей/предыдущей модели на hover.
- **Confirm на деактивацию** (`Активна → Архив`). На обратный путь confirm не нужен.
- **Vertical accent bar** (2-3px primary слева) для активного элемента списка — увеличит affordance «вы здесь».
- **Клавиатурная навигация в списке**: `↑↓` / `j k` для перехода между моделями, `Enter` открывает, `/` фокусирует поиск, `Esc` снимает выделение.

## 5. Detail · Фото (галерея)

**Что хорошо**: masonry с разной высотой плиток (органично для портретов), hover-state с `#1`-маркером и actions (expand/edit/delete), подпись + дата на градиентной ленточке, поиск по подписи (а не по имени файла), drag-to-reorder с хинтом «сохраняется автоматически», sort + layout рядом.

**Правки:**

- **Конфликт drag-reorder vs sort `По дате`**: если сорт активен и пользователь тянет — что происходит? Решение: при попытке drag в сорте `По дате` показывать toast «Чтобы менять порядок, переключите сорт на Manual» с кнопкой `Переключить`.
- **Drag-handle (⋮⋮) дублирует** возможность тянуть за саму плитку. Убрать handle, тянуть за всю плитку (cursor: grab). `#N` остаётся как индикатор позиции.
- **Hover-actions: expand vs edit/delete**: разнести. `Expand` отдельной иконкой (самое частое действие). `Edit + delete` под `…` overflow, либо delete вынести в Lightbox/детальный режим. Сейчас delete рядом с edit опасен.
- **Формат дат**: `18 апр` без года ломается на границе года. Использовать relative (`вчера`, `2 дня назад`, `12 апр`), для прошлого года — `12 апр 2025`.
- **`PHOTO 06 · 280px`** — служебный mock-плейсхолдер. В production не показывать.
- **`Masonry` toggle** без видимой альтернативы — непонятно что делает. Использовать segment control `Masonry | Grid` или dropdown `Вид: Masonry ▾`.
- **Bulk select / batch actions**: на 142 фото нужен select-mode для массового удаления / скачивания / reorder. Toggle в шапке или Cmd+click → action bar внизу.
- **Mock «без подписи»**: что показывает плитка с пустой подписью? Лента схлопывается или `Без подписи · 18 апр` muted-цветом.
- **`За месяц` фильтр**: показывать активное значение в пилюле — `За месяц: апрель 2026 ✕`.
- **Drag-hint** в шапке: после первого использования сделать dismissable (или скрывать с возможностью вернуть через `?`).

## 6. Detail · Видео

**Что хорошо**: единая 16:9 сетка (vs masonry — правильно для видео), длительность в правом нижнем углу (стандарт индустрии), play-overlay по центру, хинт про лимиты и форматы наверху, иконка `info` намекает на справку. На hover: drag-handle, edit, delete, подпись + дата.

**Правки:**

### Feature parity с фото-вкладкой

Большая асимметрия: на `Фото` есть поиск, фильтр по дате, sort, layout, drag-reorder, hover-actions из 3 кнопок. На `Видео` — только sort и hover из 2 кнопок. Решить:

- **Поиск** по названию/подписи — нужен.
- **Фильтр** по длительности (короткие/длинные) — полезен.
- **Reorder** — есть (drag-handle на hover показан), но нет `#N` маркера.

### Специфика видео

- **Добавить `expand` кнопку** в hover-actions — открывать Lightbox/fullscreen. Сейчас единственный путь — клик по play, что неявно.
- **`#N` порядковый номер** на hover (как у фото) — иначе reorder вслепую.
- **`Edit` для видео**: что редактируется (подпись, обложка, trim)? Если несколько действий — `…` меню вместо одной иконки.
- **Унифицировать hover-actions** с фото: `expand | edit | … (delete)`.
- **Mock «без обложки»** — placeholder с иконкой play и именем файла (не все видео имеют thumbnail).
- **Mock «обрабатывается / transcoding»** — после загрузки видео нужно время на конвертацию. Плитка с `Обрабатывается… 32%` или `Готовится превью`.
- **Mock empty state** (модель без видео) — `Видео ещё нет · Загрузите первое`.
- **Длительность для длинных видео**: формат `H:MM:SS` для часовых, `MM:SS` для коротких, без leading zero для часов.
- **Чип качества/размера на hover**: `1080p · 124 МБ`.
- **Аудио-индикатор**: иконка 🔇 для видео без звука.
- **`По дате`**: uploaded vs recorded — уточнить в tooltip или дать выбор.
- **Пагинация** для 24 видео: бесконечный скролл / `Показать ещё` / номера страниц — выбрать и показать.
- **Mock длинной подписи** (truncate с ellipsis + tooltip) для фото и видео.
- **Delete destructive стиль**: красный фон/иконка на hover самой кнопки delete. Сейчас она нейтральная.

## 7. Loading

**Что хорошо**: двухстрочные скелетоны имитируют реальную структуру строки (avatar + имя + мета). 6 строк покрывают viewport. Шапка списка остаётся живой — фильтры доступны.

**Правки:**

- **Согласованность счётчиков**: `Модели 9` и `Активные 7` показаны до загрузки списка. Либо все живые (cached) с muted-стилем, либо все skeleton.
- **Правая панель** во время лоада показывает обычный empty-hint. Заменить на muted «Загружается список моделей…» или убрать hint до окончания.
- **Mock `Loading · slow`**: после ~5 сек скелетоны → `Загружаем модели… ⏳`.
- **Mock `Loading · failed`**: error state с retry-кнопкой.
- **Mock `Loading · deep link`**: когда URL содержит `?model=sofia.reign`, правая панель должна показывать свой skeleton (avatar circle + lane под имя + табы).
- **Скелетоны разной ширины** (60-90% контейнера случайно) — выглядит более «натурально».
- **Race conditions**: что если пользователь применит фильтр до окончания загрузки — запрос отменяется/перезапускается. Проверить.

## 8. Upload progress

**Что хорошо**: отдельный banner в шапке галереи показывает текущий файл, общий прогресс и счётчик «N из M». Скелетоны с диагональной штриховкой имитируют загружающиеся плитки.

**Правки** (приоритет — указал пользователь):

- **Отступы и визуальная отдельность banner’а**: сейчас «прилипает» к табам и фильтрам. Решение по возрастанию вмешательства:
  1. `margin-top: 16-24px` от табов, `margin-bottom: 24px` до фильтров.
  2. `border` (1px primary-soft) или мягкий фон `primary-soft-1` — чтобы читался как «временная плашка».
  3. Sticky к верху content-area — чтобы при скролле прогресс оставался видимым (спорно, усложняет).
  4. Slide-down при появлении / slide-up при завершении.
  
  Минимум — **1 + 2**.
- **Дубль `64%`** в banner и поверх первого скелетона. Убрать со скелетона — диагональной штриховки достаточно.
- **`Загрузка фото · 4 из 6`** двусмысленно (закончил 4 или сейчас идёт 4-й?). Уточнить: `Файл 4 из 6 · studio_terrace_07.jpg`.

## 9. Drop overlay

**Что хорошо**: чёткий dashed-border вокруг drop-зоны, контент за оверлеем приглушён, карточка по центру с конкретикой `Отпустите, чтобы загрузить 3 фото` + `JPG · PNG · WEBP · до 25 МБ каждое` (формат + лимит превентивно).

**Правки:**

- **Расширить drop-zone до всей правой панели** (не только галерея). Иначе пользователи промахиваются. Если ограничено намеренно — задокументировать почему.
- **Pre-validation на drop**: если среди файлов есть невалидные — показывать `Отпустите, чтобы загрузить 2 фото · 1 файл будет пропущен (.heic)`. Не обещать загрузку всех, если часть не пройдёт.
- **Mock drop overlay для видео-вкладки** (другие форматы MP4/WEBM/MOV, лимит 500 МБ) — пока нет.
- **Конфликт drag-paradigm**: `Перетащите, чтобы изменить порядок` остаётся в шапке во время external drop. Скрывать или временно заменять на `Отпустите для загрузки`.
- **Mock «неподдерживаемый тип»**: red border + копирайт `PDF не поддерживается · принимаются JPG, PNG, WEBP`.
- **Mock «превышает лимит размера»**: `2 загрузятся · 1 превышает лимит 25 МБ`.
- **Mock «нет места» / квота storage** — если планируется учёт.
- **Тайминг fade-in оверлея в спеке**: ~150 мс, чтобы не было прыжка.
- **Pointer-events на оверлее**: `pointer-events: none` чтобы drop попадал в зону, а не в оверлей. Технический момент, проверить в живом prototype.

## 10. Lightbox

**Что хорошо**: тёмный приглушённый backdrop (видны силуэты UI), `‹ ›` навигация в полупрозрачных чипах, `×` в правом верхнем, подпись + мета внизу, счётчик `03 / 24`, keyboard-hint `← →` / `Esc`.

**Правки:**

- **Toolbar действий отсутствует**. Добавить в правый верхний угол рядом с `×`:
  - Скачать
  - Редактировать (подпись)
  - `…` overflow → `Установить как аватар`, `Скопировать ссылку`, `Удалить`
  
  Сейчас Lightbox read-only, для удаления нужно закрыть и найти плитку — лишнее.
- **Формат даты**: `2025-04-12` (ISO) → `12 апр 2025` (human-readable). Согласуется с подписью «апрель · крыша».
- **Разрешение в мете**: добавить `1440 × 1920 · JPG · 2.4 МБ`.
- **Граничные состояния навигации** (1/24 и 24/24): cyclic vs disabled — выбрать политику и показать в моках.
- **Zoom**: click для 2× или pinch на тач — особенно для портретов 1920px.
- **Mock длинной подписи** (multi-line) — не сломает ли вёрстку центрирования меты.
- **Упростить keyboard-hint**: одна строка `← → навигация · Esc закрыть` вместо chip-ов с иконками.
- **Контраст mono-меты**: проверить WCAG AA (4.5:1) на тёмно-сером фоне.
- **Backdrop opacity**: 80-90% чёрный + лёгкий blur (2-4px) — стандарт. Сейчас читается как «слегка приглушённый», проверить в production с яркими фонами.
- **Loading state в lightbox**: skeleton или blurred thumbnail пока крупное фото грузится.
- **Mock Lightbox · видео**: нативный плеер (play/pause, seek, volume, fullscreen). Сейчас mock только для фото.

## 11. Video error

**Что хорошо**: inline error-tile там же, где ожидалась плитка. Конкретика (`studio_apr_long.mov · 612 МБ`), обе причины (размер + формат) сразу, рекомендация-действие (`Сожмите в .mp4 (H.264)`), две кнопки (`Повторить` / `Отменить`), backup toast снизу, мягкий розовый цвет ошибки без агрессии.

**Правки:**

- **Цвет toast’а ≠ inline-error**: плитка розовая (destructive-soft), toast lavender. Если оба — error, должны быть одного destructive-tone. Если разные уровни (error vs warning) — задокументировать разницу.
- **Дедупликация toast vs inline**: показывать toast только если плитка вне viewport. Иначе на экране две одинаковые ошибки. С toast’а — кнопка `Перейти` (скроллит к плитке).
- **`Повторить` для validation-fail бессмыслен** — файл всё ещё .mov 612 МБ. Заменить на `Заменить файл` (открыть file picker). `Повторить` оставить только для network-fail.
- **`Отменить`** двусмысленно (отменить upload? убрать плитку?). Переименовать в `Удалить` или `Закрыть`.
- **Mock узкого viewport**: текст ошибки на узком экране должен truncate с tooltip’ом или раскрывать modal с подробностями.
- **Выделение проблемных значений**: и `612 МБ`, и `.mov` одним destructive-цветом. Рекомендацию `.mp4 (H.264)` — позитивным (зелёный/синий акцент).
- **A11y**: toast → `aria-live="assertive"`, plitka → `role="alert"`.
- **Mock множественных ошибок**: при 2+ failed — агрегированный toast «2 видео не загружены · подробнее», не спам.
- **Persistent error toast** (без auto-dismiss) — пользователь сам решает закрыть.
- **Mock warning state** (отличается от error) — например, «низкое разрешение, продолжить?». Сейчас только error.
- **Структура длинного error-текста** в bullets:
  - Размер: 612 МБ (лимит 500 МБ)
  - Формат: .mov (нужен .mp4 H.264)
- **Анимация удаления error-плитки**: fade-out + плавный reflow сетки.
- **Drag-handle на error-плитке** не нужен — reorder для не загруженного файла бессмыслен.
- **Mock retry-fail**: после починки одной причины из двух — текст ошибки обновляется только с оставшейся.

## Сквозные темы

### Микрокопия

- **Плюрализация**: проверить везде `1/2-4/5+` формы (`файл / файла / файлов`, `видео`, `фото`).
- **Даты**: единый формат — human-readable (`12 апр 2025`), с relative для свежих (`вчера`, `2 дня назад`). Убрать ISO (`2025-04-12`).
- **Negative hints** переписать в позитив (`Markdown не поддерживается` → `Plain text. Перенос строки сохраняется`).

### Состояния

Для каждой вкладки нужен полный набор моков:

- `Empty · zero` ✓
- `Empty · filter` ✓ (но добавить `Empty · search`)
- `Loading` ✓ (но добавить `Loading · slow`, `Loading · failed`, `Loading · deep link`)
- `Error` (network, auth, permission) — отсутствует на уровне страницы
- `Validation error` в форме создания — отсутствует
- `Empty · video` (отдельно от фото) — отсутствует

### Accessibility

- Контрастность mono-текста в Lightbox (WCAG AA).
- `aria-live` для toast-ов и `role="alert"` для inline-error плиток.
- Keyboard navigation в списке моделей (`↑↓` / `j k` / `Enter` / `Esc` / `/`).
- Tab-order в slide-out форме создания.

### Согласованность

- **Hover-actions** на плитках Фото и Видео — унифицировать (`expand | edit | … (delete)`).
- **Drag-handle** — везде или нигде. Сейчас на фото — раздельный handle, на видео — тоже handle. Единый паттерн: тянуть за плитку, handle убрать (или оставить как индикатор).
- **Цвет destructive-action** (delete) на hover — везде в красный, не нейтральный.
- **CAPS LOCK на label’ах** в форме vs sentence case в остальных местах — выровнять.
- **Счётчики** (количество файлов) — один источник истины. Сейчас три места одновременно.

### Edge cases

- Длинные строки (имена, подписи, названия файлов, агентств) — везде нужны mock’и truncate + tooltip.
- Узкие viewport’ы — поведение split-вью на <1280px.
- Tach-устройства — pinch-zoom в Lightbox, hover-actions через long-press.
- Cyclic vs disabled навигация на границах (1/N и N/N).
