### R2.2. Поля профиля: affordance редактируемости

Поля в карточке `Поля профиля` выглядят как обычный текст, хотя являются inline-редактируемыми. Это неочевидно:

- `Name * / Sofia Reign` — plain text, никакой рамки.
- `Platform * / OnlyFans ▾` — chevron намекает на dropdown, но рамки/фона нет.
- `Agency * / Aurora Media ▾` — то же.
- `Tableau ID / 🔒 TBL-2351 · read-only · из Tableau` — иконка замка чётко говорит «нельзя» (это работает).

Проблемы:
1. Поля выглядят как текст → пользователь не понимает, что они интерактивные.
2. Chevron-аффорданс непоследователен: есть на Platform/Agency, нет на Name/Alias — хотя оба редактируемы.
3. Хинт `click → inline edit` сверху требует читать инструкцию. UI должен подсказывать структурой, а не пояснениями.

**Решение — Вариант A (субтильные input-borders по умолчанию):**

- Каждое editable поле имеет лёгкую рамку: `1px solid muted` либо `border-bottom: 1px muted`. Достаточно тонкую, чтобы карточка не стала «формой», но видную как affordance.
- На hover: рамка темнеет до `border` (стандартный токен), `cursor: text`.
- На focus/edit: полноценный border с primary (`ring`-стиль focus).
- **Chevron**: остаётся у dropdown-полей (Platform, Agency), убрать у text-полей (Name, Alias).
- **Read-only поля** (Tableau ID): без рамки, иконка-замок остаётся. Контраст уже работает — только усилить за счёт того, что остальные поля стали явными input’ами.
- **Хинт `click → inline edit` убрать** — становится избыточен, когда поля выглядят как поля.

### R2.4. Detail · Фото — drag-handle: решение

Решение продукта: **оставить как сделано в R2 — без handle, тянуть за всю плитку**.

**Why**: меньше визуального шума на hover, естественный паттерн (Pinterest/Figma-style), `#N` + hint наверху уже сообщают affordance.

**How to apply**: при hover на плитку cursor меняется на `grab`, при начале drag — `grabbing`. Это критично — без явного cursor change пользователь не поймёт что плитка draggable.

**Известный риск**: конфликт click (открыть Lightbox) vs drag (reorder) на медленном тачпаде. Если по итогам тестирования на реальных пользователях этот риск проявится — вернуться к Варианту B (handle на hover) или C (drag только в режиме `Выбрать` / `Вручную`).

### R2.8. Confirm-диалог при закрытии slide-out

**Что починили**:
- ✅ Confirm-диалог при попытке закрыть форму с несохранённым вводом (R1.3).

#### R2.8.1. Терминология

Заголовок slide-out за модалкой — `Новая модель`. Должно быть `Новый клиент` (правило R0).

#### R2.8.2. Визуальный приоритет destructive vs safe

Сейчас destructive (`Закрыть без сохранения`) визуально тяжелее safe (`Продолжить ввод`). Eye-catch падает на destructive, пользователь чаще будет терять ввод по инерции.

**Решение** (рекомендую):
- `Продолжить ввод` — primary filled (синий, как основной CTA в системе).
- `Закрыть без сохранения` — ghost/outlined red text без primary-веса.

Альтернатива: `Продолжить ввод` outlined neutral, `Закрыть без сохранения` filled red destructive — оба явные, но safe по умолчанию primary.

#### R2.8.3. Default focus / Enter / Esc behaviour

Зафиксировать в спеке:
- Open dialog → focus на `Продолжить ввод`.
- `Enter` → triggers focused (safe) action.
- `Esc` → закрывает диалог, возвращает в форму.
- Destructive — только через Tab + Enter или клик.

#### R2.8.4. Click outside поведение

Диалог поверх slide-out. Click на overlay вне диалога — игнорируется (только Esc или явная кнопка). Описать в спеке.

#### R2.8.5. Undo для случайного закрытия (опционально)

После `Закрыть без сохранения` — toast `Форма закрыта · Восстановить` на 5-10 сек. Не блокирующее, улучшает forgiveness.

#### R2.8.6. Trigger «грязной» формы

Уточнить: dirty по `touched` (изменено хотя бы одно поле) или `valued` (есть значение ≠ default). Если пользователь только открыл форму и не трогал поля — confirm не нужен, закрывать сразу.

- **Auth error** (403): `Войти заново` (переход к login, текущая попытка сохраняется).

#### R2.15.8. Усиление цифры превышения

`612 МБ (лимит 500 МБ)` — пользователь сам считает превышение. Усилить:

`612 МБ · превышено на 112 МБ` или `612 МБ из 500 МБ (лимит)`.

Опционально, не критично.

#### R2.15.9. `Заменить файл` с иконкой / `Закрыть` без — несогласованность

Иконки на кнопках: `🔄 Заменить файл` есть, `Закрыть` без иконки. Либо обе с иконкой (`✕ Закрыть`), либо обе без. Иконки на secondary-кнопках обычно опускают — рекомендую **убрать иконку у `Заменить файл`**, оставить чистый текст.

#### R2.15.10. Accessibility

Inline-error: `role="alert"` + `aria-live="polite"` чтобы screen reader проговорил.

Toast: тот же `role="alert"`, плюс `aria-live="assertive"` для немедленной озвучки.

Color contrast: pink-50 фон + red-700 текст — проверить 4.5:1. На скрине читается, но формальная проверка нужна.

#### R2.15.11. State persistence

Refresh страницы → error-state должен исчезать (in-memory, не persistent в БД). Зафиксировать.

Если хотим persistent (например, очередь невалидных файлов восстанавливается) — отдельная фича, scope-cut на R3+.

#### R2.15.12. `Перейти к плитке` — условный CTA

Кнопка в toast'е активна всегда. Логика:
- Inline-tile в viewport (visible на экране) → кнопка `Перейти к плитке` бесполезна.
- Если в viewport — заменить кнопку на `Закрыть` или скрыть toast вообще (R2.15.3).

#### R2.15.13. Tab counter `Видео 24` после неудачи

После failed upload счётчик НЕ инкрементится (24 → 24). Корректно. Зафиксировать в спеке.

# Handoff в Claude Design

Этот раздел — самодостаточный action-list. Если читаешь его впервые — начни **отсюда**, журнал R2.0–R2.15 выше нужен только за деталями.

## TL;DR — топ-10 правок (применить первыми)

1. **Терминология «клиент вместо модели»** — пакетно по всем экранам. Не точечно. Таблица переводов в [subplan-3-clients.md, раздел 0](subplan-3-clients.md). На R2 не применена в 6+ местах: page title, CTAs (`+ Добавить`, `Создать`), empty-карточки, slide-out title, detail-empty копирайт.

2. **CAPS LOCK на label'ах формы** (R2.11.2) — sentence case + muted gray. `ИМЯ` → `Имя`, и так все 6 полей slide-out.

3. **Три состояния filter-chips** (R2.0a.4): `default / value-set / changed-by-user`. Сейчас `Платформа все` (нейтрал) и `Активные 0` (со счётчиком) выглядят одинаково. Только `changed-by-user` выделяем синим.

4. **Активный элемент в master-list — оставить только vertical accent bar** (R2.1). Background fill и border убрать — три сигнала одновременно перебор.

5. **Поля профиля — input-borders по умолчанию** (R2.2). Plain text без рамки выглядит как неинтерактивный — пользователь не понимает что можно редактировать.

6. **Detail-панель в empty-states** (R2.0.4, R2.0a.6): схлопнуть на zero и filter empty. Master на всю ширину — двойной empty-state бесполезен.

7. **Counter `01 / 24` в Lightbox** (R2.10.2, R2.14.2): вынести в шапку lightbox из meta-строки. Это главный нав-сигнал при swipe.

8. **CTA `Продолжить ввод` в close-confirm — primary, `Закрыть без сохранения` — ghost-red** (R2.8.2). Сейчас destructive визуально тяжелее safe — пользователь будет терять ввод по инерции.

9. **Upload progress banner — белый фон + blue border** (R2.9.1). Track сливается с фоном, прогресс не читается.

10. **Hint и error по `MOV` в Video** (R2.15.2): hint обещает «MOV ОК», error отвергает. Уточнить: `MOV (только кодек H.264)`.

## Пакетные правила (применить ко всем экранам Subplan 3)

### Терминология
Пройти таблицу переводов [subplan-3-clients.md, раздел 0](subplan-3-clients.md) **подряд по всем экранам**. На R2 это правило применено частично — после следующего круга должно быть 100%.

### CTA с сущностью
- `+ Добавить` → `+ Добавить клиента` (везде кроме списка-с-данными, где контекст уже задан).
- `Создать` → `Создать клиента` (slide-out submit).
- `Сохранить`, `Отмена`, `Закрыть` — без сущности, ОК.

### Tooltip на all иконках
В lightbox top-right toolbar (`download · pencil · ?user · … · close`) и везде где есть icon-only buttons — `title=` или tooltip-on-hover обязателен.

### Состояния filter-chips
Три явных состояния (см. TL;DR-3). Применимо к chip'ам везде, где они есть (master filter, possible future filters в Detail).

### Loading thresholds
- <300мс — без skeleton.
- 300мс-2с — skeleton.
- 2-15с — skeleton + `Загружается …` сообщение.
- >15с — error fallback с retry.

### Skeleton структура
Skeleton подражает реальной структуре (R2.12.5): круглый avatar (не квадрат), tab-row, content-grid. Не «3 абстрактные line'ы» — иначе layout прыгнет при появлении контента.

### Hover affordance
Cursor change обязателен для всего, что не очевидно интерактивное:
- Plain-text editable поля → `cursor: text` на hover.
- Draggable tiles → `cursor: grab` на hover, `grabbing` при drag (R2.4).
- Disabled actions → `cursor: not-allowed`.
