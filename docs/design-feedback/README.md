# Design Feedback — workflow

Рабочий процесс для ревью дизайна каждого крупного раздела CRM (Subplan).

## Цель

Структурированный двусторонний обмен между сессией с Claude Design (генерация макетов) и сессией ревью (этот процесс). Один документ на Subplan фиксирует все правки, согласованные решения и открытые вопросы — так Claude Design получает консистентный brief без потерь контекста.

## Шаги

1. **Claude Design генерирует макеты** по Subplan’у (List, Detail, Empty-states, Loading, Error и т.д. — в Figma или другом инструменте).
2. **Скриншоты по очереди** — экран за экраном, состояние за состоянием. Не все сразу: один скрин = одна порция фидбэка, чтобы внимание не размазывалось.
3. **Per-screen фидбэк** в формате:
   - **Что хорошо** — фиксируем удачные решения, чтобы их не сломали при доработке.
   - **Правки** — буллеты с конкретикой (что не так + предлагаемое решение).
4. **Консолидация** в `docs/design-feedback/subplan-N-{name}.md` (см. шаблон ниже).
5. **Открытые вопросы** в конце документа — ответы от продукта/владельца требуются до доработки макетов. Без ответов часть правок останется слепыми.
6. **Передача обратно в Claude Design** — открыть новую сессию, сослаться на файл: «Прочитай `docs/design-feedback/subplan-N-{name}.md` — фидбэк по макетам, доработай по этому списку».

## Конвенция именования

`docs/design-feedback/subplan-{N}-{name}.md`

- `N` — номер subplan’а (`1`, `2`, `3`, ...).
- `name` — короткое английское имя секции (`clients`, `agencies`, `operators`, ...).

Примеры: `subplan-3-clients.md`, `subplan-4-agencies.md`.

## Шаблон документа

```markdown
# Subplan N · {Section Name} — фидбэк по дизайну

**Контекст для Claude Design**: ниже — структурированный фидбэк по макетам Subplan N. Просмотрены все вкладки/состояния: [перечислить].

Общая оценка: [одно-два предложения].

---

## 1. {Screen / State name}

**Что хорошо**: [bullet или абзац].

**Правки:**
- [bullet с конкретикой и предлагаемым решением]
- ...

---

## 2. {Next screen}
...

---

## Сквозные темы

### Микрокопия
### Состояния (см. чеклист ниже)
### Accessibility
### Согласованность
### Edge cases

### Открытые вопросы (нужны решения от продукта)
1. [вопрос]
2. ...
```

## Чеклист состояний

Для каждого экрана нужны mock’и всех релевантных состояний:

- [ ] **Empty · zero** — данных никогда не было.
- [ ] **Empty · filter** — данные есть, но под фильтр не подходят.
- [ ] **Empty · search** — отдельно от filter, если поиск независимый.
- [ ] **Loading** — стандартные скелетоны.
- [ ] **Loading · slow** — после 5+ сек, альтернативный копирайт.
- [ ] **Loading · failed** — error с retry.
- [ ] **Loading · deep link** — когда URL уже содержит сущность (отдельный skeleton для detail-панели).
- [ ] **Error · network / permission** — на уровне страницы.
- [ ] **Validation error** — в формах.
- [ ] **Success / confirmation** — после действий.
- [ ] **Hover state** — для интерактивных элементов с actions.
- [ ] **Disabled / read-only** — где применимо.
- [ ] **Long content** — длинные имена / подписи / списки (truncate, multiline).
- [ ] **Narrow viewport** — поведение на <1280px / split-режим.

## Сквозные правила (накопительный список)

Этот раздел заполняется по мере ревью subplans — фиксированные решения, которые Claude Design должен соблюдать во всех новых макетах.

> **Cross-cutting правила**: см. `_cross-cutting.md` (создаётся после 2-3 ревью).

## Артефакты

- `subplan-N-{name}.md` — фидбэк по конкретной секции.
- `_cross-cutting.md` — сквозные паттерны и решения, действующие на все секции (накопительный документ).
- `_decisions.md` — реестр ответов на открытые вопросы (что решено, кем, когда).
